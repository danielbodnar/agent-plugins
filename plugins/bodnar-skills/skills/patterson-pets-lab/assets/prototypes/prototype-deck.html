The order is deliberate. Surfaces first, so you know which environment to reach for. Configuration second, so the agent knows your code. Use third, build fourth, constrain fifth, deploy sixth. Each section sets up the next.

01

Introduction

8 minutes · orientation

An AI coding agent is the same kind of model, in different rooms. The room changes what is easy, what is hard, and what is dangerous. Pick the room before you pick the tool.

The five surfaces

Every coding agent in production today lives in one of these. The model behind them is often the same. The trade-offs are not.

01
Browser / Chatclaude.ai · ChatGPT · v0 · bolt.new
Stateless threads. Zero setup. Best for research, drafts, and questions you can throw away.No filesystem. Copy-paste tax. Cannot run your code.
02
Terminal / CLIClaude Code · OpenCode · Codex · Goose
Full repo as context. Scriptable. The agent reads your AGENTS.md, runs your tests, opens your files.CLI literacy expected. No GUI affordances.
03
IDE pluginCursor · Copilot · Cline · Windsurf · Zed
Inline edits. Autocomplete. The current buffer is the context.Bound to the editor. Narrow context. Easy to over-trust.
04
CI/CD pipelineGH Actions agents · scheduled bots · PR reviewers
Triggered by events. Runs without humans. Audit trail by default.No human in the loop. Demands strong guardrails.
05
Desktop / Computer UseComputer Use · Antigravity · Operator
Can drive any application. UI tests, cross-app workflows, browser automation.Slow. Fragile. The largest security surface.

Pick by task, not by tool

The skill is not picking a favorite. It is noticing when the surface and the task do not match.

02

Foundation

10 minutes · lesson/foundation

An agent without instructions is a chat window with file access. The repo is where you teach it who you are and how you work.

Every agent is three layers

The names change between vendors. The structure does not.

Instructions — who the agent is, how it should behave, what your codebase cares about. Persistent across sessions. Lives in AGENTS.md, CLAUDE.md, .cursorrules, or .github/copilot-instructions.md.

Tools — what the agent can do. Read files, run commands, hit APIs. Granted explicitly, scoped narrowly. Built-in tools, MCP servers, skills.

Context — what the agent can see right now. Open files, current selection, recent commands, retrieved docs. Mentioned with @, supplied by the harness, or fetched on demand.

AGENTS.md, in practice

This is the file most agents read first. Keep it short. Keep it specific. Update it when your conventions change.

AGENTS.md
# Project: payments-api

Stack: Bun + TypeScript + Vitest
Style: strict, no any, named exports
Tests: colocated as *.test.ts

## Conventions
- All scripts use #!/usr/bin/env bun
- Prefer Bun.$ over spawnSync
- Lint with bun run lint before commit

## Forbidden
- Editing /migrations without approval
- Direct production deploys

Slash commands as repeatable workflows

If you find yourself typing the same prompt twice, write it down once. Custom verbs live in .claude/commands/ or your agent's equivalent.

.claude/commands/review.md
---
description: Review staged changes before commit
---

Run git diff --staged, then for each file:
1. Check it matches AGENTS.md conventions
2. Flag any TODO/FIXME left behind
3. Suggest a conventional commit message

Output format: markdown summary with inline diffs.
hands-on · 4 minutes
Edit AGENTS.md in your fork. Add a /test-this slash command that runs the test suite for the file currently open. Verify by asking your agent to use it.
The /review command you wrote here is the same one we will wire into CI in §6. The lab is one long setup, structured as six short ones.
03

Using Agents, Skills & MCP Servers

10 minutes · lesson/skills-and-mcp

Skills teach an agent what to do. MCP servers grant an agent things to do it with. One is procedural knowledge. The other is live tools and data.

Skills, from skills.sh

A skill is a folder with a SKILL.md and any supporting files. Your agent loads it on demand. The same npx skills add command works across nineteen different agents.

~/lab
$ npx skills search "pdf"
→ anthropics/skills/pdf      ★ 95.7K
→ anthropics/skills/pdf-reading

$ npx skills add anthropics/skills/pdf
  installed → .agents/skills/pdf/
  registered with claude-code

MCP servers — tools and data, not knowledge

The Model Context Protocol is the standard for handing agents access to live systems. Filesystems, databases, ticket trackers, search indexes, your build system.

.mcp.json
{
  "mcpServers": {
    "filesystem": {
      "command": "bunx",
      "args": [
        "@modelcontextprotocol/server-filesystem",
        "/Users/me/lab"
      ]
    },
    "github": {
      "command": "bunx",
      "args": ["@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "$GH_PAT" }
    }
  }
}
hands-on · 4 minutes
Install one skill from skills.sh that matches your daily work. Add a filesystem MCP server scoped to ~/lab. Restart your agent. Ask it to do something neither could do before.
04

Building Your Own Agents, Skills & MCP Servers

7 minutes · lesson/your-skill

In §3 you consumed. Now you produce. The senior engineer's tacit knowledge becomes a file the agent reads. Onboarding goes from weeks to a single npx skills add.

Anatomy of a team skill

A team skill captures the things every developer at Patterson should know. Conventions, banned patterns, internal libraries. Publish once, install everywhere.

patterson/skills/house-style/SKILL.md
---
name: patterson-house-style
description: Use this when writing code in any
  patterson/* repo. Covers our TypeScript conventions,
  internal libs, and review checklist.
---

# House style

## Imports
- Use named exports
- Internal pkgs: @patterson/*
- Forbidden: lodash, moment, axios

## Tests
- Vitest, colocated as *.test.ts
- Snapshot tests forbidden in /core

Publish, install, trigger

~/patterson-skills
# 1. Push to a repo (yours, the team's, doesn't matter)
$ git push origin main

# 2. Anyone on the team installs it
$ npx skills add patterson/skills/house-style

# 3. Or wire it into the team bootstrap script.
#    A new hire runs ONE command and has full context.
hands-on · 5 minutes
Write a ten-line skill capturing one rule you actually care about. Push it. Install it back. Trigger it in a fresh session. The point is not the skill, it is the publishing loop.
A small MCP server follows the same shape: a directory, a manifest, a TypeScript file that exports tool definitions. The same .mcp.json from §3 wires it in. We will not build one in the lab, but the path from skill to MCP server is shorter than it looks.
05

Guardrails & Quality Control

5 minutes · lesson/guardrails

An agent that can do anything will, eventually, do something you did not want. The fix is structural, not hopeful.

Two layers: permissions and hooks

Permissions are the door — explicit allow and deny lists with glob patterns. The agent asks before running anything off-list.

Hooks are the safety net behind the door — shell scripts that run at lifecycle points and block actions by exiting non-zero. Same mental model as a git hook.

.claude/settings.json
{
  "permissions": {
    "allow": [
      "Read(*)",
      "Bash(bun *)",
      "Bash(git diff:*)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(curl * | bash)",
      "Write(/etc/**)"
    ]
  }
}
.claude/hooks/pre-tool-use.sh
#!/usr/bin/env bash
# Block any command with --force
if echo "$TOOL_INPUT" | grep -qE "\-\-force"; then
  echo "BLOCKED: --force flag" >&2
  exit 2  # non-zero = block
fi
In your terminal, you have Ctrl-C. In CI, you do not. That is why guardrails come right before §6 — once the agent runs without you watching, this layer carries the load.
06

CI/CD & Automation

8 minutes · lesson/automation

Same agent. Same AGENTS.md. Same /review slash command. Different trigger. The agent works on pull requests, issues, and cron schedules. You do not watch.

The reveal

The /review command you wrote in §2 works in CI without modification. Same instructions, same standards. The only thing that changes is the trigger.

.github/workflows/agent-review.yml
name: Agent code review
on:
  pull_request:
    types: [opened, synchronize]

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: anthropics/claude-code-action@v1
        with:
          prompt: "/review"
          claude_args: "--allowed-tools Bash(git:*)"
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}

Patterns that repay the setup

Code review on every PR. The agent diffs the PR against your AGENTS.md and comments inline.

Issue triage. Bug reports come in. The agent labels them, dedupes against existing issues, drafts a reproduction.

Scheduled maintenance. Dependency bumps, lockfile updates, doc regeneration on a cron. The agent opens PRs for human review.

The permissions allowlist from §5 is what makes this safe. In a terminal you would notice if the agent tried something destructive. In CI, the allowlist is the only thing between you and an automated mistake on main.

What to take home

Pick one repo at work. Add an AGENTS.md. Add a CI review. Watch how the next three pull requests change. That is the smallest possible step that compounds.
