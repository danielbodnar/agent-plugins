The workshop in brief

A working agent, configured against a real codebase, in one sitting.

Attendees clone a prepared repository, configure an AI coding agent against a real codebase, install community skills, wire up MCP servers, and leave with a setup they can use the same afternoon.

The tooling is vendor-agnostic and built around the concerns Patterson actually has: data handling, auditability, and avoiding lock-in to any single vendor's stack.

Why this exists

Three observations shaped the content.

01

The cost structure is shifting

Per-seat AI products are moving toward usage-based pricing. Developers who understand how to work with vendor-agnostic agents will be substantially less expensive to support than those tied to a single vendor's stack.

02

The skill gap is wider than it looks

Most developers have used a chat-based AI tool, but far fewer have configured an agent against their own codebase, written a reusable skill, or integrated an agent into code review or CI. That is where the meaningful leverage sits.

03

Enterprise concerns deserve technical answers

Data handling, auditability, credential exposure, and prompt injection are real risks. The architectural patterns taught here address those risks structurally, through how the agent is configured and constrained.

The shared project · interactive

One game. Everyone ships a module.

The hands-on work centers on a real-time multiplayer game styled after the Chrome offline runner, reskinned with pets to fit Patterson Vet and ACT's domain. The infrastructure runs on Cloudflare Durable Objects and WebSockets, provisioned ahead of time.

Each attendee builds one modular piece. At the end of the session, every module is wired together into a single working game.

Optional follow-on for teams

Each team trains a pet.

For teams that want a recurring reason to refine how they work with AI agents, the workshop includes an optional follow-on competition that begins after TechDays. Each team maintains a small GitHub repository containing a single Claude skill, which represents their team's "pet."

Every month, every pet attempts the same engineering challenge. It opens a pull request against a shared challenge repository, fields code review from a judge skill, and is either merged or rejected on the merits of its work.

Genuinely optional. The workshop stands on its own.
Code quality first, token efficiency second.

Who should attend

Designed for developers who haven't configured an agent yet.

The workshop is a strong fit for senior engineers who want to shape how their team adopts AI tooling, and for anyone who has used AI tools casually but wants to move past chat-window workflows into something more integrated with day-to-day work.

Engineering managers, product managers, and designers interested in AI-assisted development are equally welcome.

  • Curious developers Used a chat AI tool. Haven't configured an agent against their own codebase yet.
  • Senior engineers shaping team adoption Want to influence how their team picks up AI tooling without being prescriptive.
  • Engineering managers planning adoption Need to understand the workflow side, not just the policy side.
  • Product & design folks Curious about the workflow side of AI-assisted development.
  • Anyone joining a team's competition entry Even on the non-technical side, the session provides the context to contribute.
